Based on the onboarding module you've implemented, here's a breakdown of the authorization mechanisms used:
Authorization in the Onboarding System

    JWT-based Session Tokens:
        Generated when a user initiates the onboarding process
        Used to track and authenticate all subsequent onboarding steps
        Carries the session ID and user email in its payload
        Expires after 7 days (configured via SESSION_TOKEN_EXPIRE_DAYS)

    Bearer Token Authentication:
        Required for all API endpoints except initialization
        Format: Authorization: Bearer <jwt_token>
        Extracted and validated by the get_onboarding_session dependency

    Token Validation Process:
        JWT is decoded using the SECRET_KEY environment variable
        Session ID is extracted and verified against the database
        Ensures the session exists and hasn't expired

    Session Verification Flow:
        JWT is first verified cryptographically
        Then the session is validated against the onboarding_progress collection in AstraDB
        Both checks must pass to access protected endpoints

All this is implemented through:

    The create_session_token function to generate tokens
    The decode_session_token function to validate tokens
    The get_onboarding_session dependency to enforce authentication

The system creates a stateful authentication mechanism that allows users to start onboarding before they have a full account, while maintaining security throughout the multi-step process.

def create_access_token(data: dict, expires_delta: Optional[timedelta] = None):
    to_encode = data.copy()
    if expires_delta:
        expire = datetime.utcnow() + expires_delta
    else:
        expire = datetime.utcnow() + timedelta(minutes=15)
    to_encode.update({"exp": expire})
